# 수집함(Inbox) 자동 분류 및 병합 워크플로우

사용자가 `/process-inbox` 명령어 또는 "수집함 정리해줘"라고 요청했을 때 실행하는 워크플로우입니다.
에이전트(AI)는 수집함에 쌓인 `raw_dumps` 데이터를 읽고, 기존 DB 내역과 똑똑하게 비교(RAG)하여 중복을 방지하고 내용을 병합합니다.

## 실행 절차 (AI 행동 지침)

### Step 1: 현재 데이터베이스 읽기 (Context Gathering)
먼저 현재 보관되어 있는 모든 기획 및 아이디어 현황을 파악해야 합니다.
1. `j:\AI\2인 게임 관리 툴\app\data\ideas.json`, `meetings.json`, `tasks.json` 파일을 읽어서 기존 리스트를 분석합니다. (앱의 양방향 동기화(2-Way Sync)로 인해 로컬 JSON은 항상 최신 DB와 동일한 상태입니다)
2. 가장 중요한 `data/raw_dumps.json` 파일을 열어 처리해야 할 러프 메모(Inbox) 데이터를 확인합니다.

### Step 2: RAG 기반 비교 분석 (Comparison)
수집함에 있는 원본 텍스트 내용을 차례로 하나씩 분석하며 다음 3가지 중 하나로 분류 전략을 세웁니다.

- **[Case A: 병합(Merge) / 업데이트]**
  - 기존 리스트(ideas.json)에 이미 존재하는 아이디어나 매우 유사한 기능일 경우.
  - 별도의 새로운 카드를 생성하지 **않고**, 기존 아이디어의 `description` 항목이나 코멘트로 내용을 덧붙여 보강합니다. (예: `\n\n**[추가 논의 사항]** ~내용~`)
- **[Case B: 충돌 및 검토(Conflict/Review)]**
  - 기존에 확정(approved)된 아이디어와 방향성이 정면으로 충돌하거나, 기존 기획을 뒤엎는 내용일 경우.
  - 기존 아이디어에 병합하되, 해당 카드의 상태(status)를 `reviewing`으로 강등시키고, `description` 상단에 주의 사항을 추가하거나, 완전히 새로운 아이디어 카드로 분리하고 `[충돌 검토 필요]` 태그를 붙입니다.
- **[Case C: 신규 생성(Create)]**
  - 기존 데이터베이스 어디에도 속하지 않는 완전히 새로운 시스템이나 기획 정보일 경우.
  - 적절한 카테고리(`게임시스템`, `세계관`, `비전`, `UI`, `기술` 등)와 태그를 요약 추출하여 데이터베이스에 새로 인서트합니다.

### Step 3: 데이터베이스 반영 스크립트 실행 (Execution)
분류 전략이 세워졌다면 이를 실제 프로젝트 데이터베이스(Supabase)에 반영해야 합니다.

// turbo
1. Node.js 스크립트 작성
   `app/tmp/` 폴더 등에 에이전트가 직접 `@supabase/supabase-js` 모듈을 사용하는 일회성 스크립트(`upsert-inbox.js` 등)를 생성합니다. (`.env.local`의 데이터베이스 접속 키 사용)
   단, Supabase 통신 여건이 여의치 않으면 단순히 로컬 `.json` 파일들을 직접 수정(fs.writeFileSync)하여 저장해도, Next.js 백그라운드 2-Way Sync가 알아서 서버에 올려주므로 이 방식을 써도 안전합니다.
   
2. 데이터 갱신 및 청소
   분석한 업데이트, 신규 생성(ideas, meetings 등)의 쿼리를 먼저 실행하고, 처리가 끝난 `raw_dumps`의 row 및 데이터는 확실하게 `DELETE` 처리하여 수집함을 깨끗이 비웁니다(Inbox Zero).

### Step 4: 완료 보고 (Report)
모든 처리가 끝나면 사용자(팀원)가 이해하기 쉽도록 무엇이 덧붙여졌고(Updates), 무슨 문서가 새로 생겼으며(Creates), 혹시 충돌된 부분은 없었는지 종합 요약하여 답변합니다.
